# Use Cases

DAD-M fits best when work spans multiple steps, produces visible artifacts, and benefits from reviewable outputs.

## Good fit

- software development across several milestones
- system analysis before implementation work
- documentation-heavy change work
- automation workflows with clear boundaries
- project planning that needs explicit deliverables
- research or analysis tasks that benefit from reusable artifacts
- learning or explanation workflows where validation, transfer, and visible uncertainty matter

## Less suitable

- one-off questions with no follow-up work
- very small tasks where a full cycle would add unnecessary overhead
- situations where no artifacts or review gates are needed

## Why these use cases fit

The method is strongest when teams need:

- traceable decisions
- strict separation between planning and execution
- repeatable output formats
- a controlled way to iterate with AI support

One early public variant in this repository is DAD-M Education. It applies the same cycle to learning and knowledge-building tasks, with stronger emphasis on validation before explanation.
